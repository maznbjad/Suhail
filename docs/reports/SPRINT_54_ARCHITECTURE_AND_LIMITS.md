# Sprint 54 — البنية والحدود قبل الإنتاج

## ما أصبحت الحزمة قادرة عليه

- تشغيل رحلة طالب كاملة محليًا.
- اختيار القدرات والتحصيلي معًا.
- تطبيق علمي/أدبي على القدرات فقط.
- تشغيل قياسين مستقلين وحفظ نتيجتين مستقلتين؛ API يخفي الإجابات ويصحح جلسات القياس على الخادم.
- إظهار ستريك بنار SVG وساعة رملية.
- تشغيل تنقل Main/Summary/Exam/Auth بقواعد واضحة.
- إدارة إعدادات عميقة محليًا ونشرها إلى API محمي.
- تسجيل حسابات آمنة عبر API.
- تخزين النشاط والقياسات والصداقات والتحديات على خادم محلي.
- تنفيذ فحوص تلقائية للبنية والواجهة وAPI.

## الحدود التي ما زالت قائمة

### 1. Streamlit ليس عميل App Store النهائي

الحزمة الحالية واجهة ويب داخل Streamlit. طبقة Sprint 54 مستقرة أكثر، لكنها لا تلغي الحاجة إلى بناء شاشات React Native + Expo الأصلية. رفع WebView فقط لن يحقق أفضل تجربة أو أداء أو تكامل إشعارات.

### 2. ملف `app.py` ما زال كبيرًا

الكود القديم موجود للتوافق مع المحتوى والملخصات والأسئلة. Sprint 54 يملك الواجهة النهائية لكنه لا يحذف كل التاريخ. قبل الإنتاج يفضّل نقل كل مسار إلى وحدات API وشاشات Expo ثم تقاعد الطبقات القديمة بدل استمرار تكديسها.

### 3. التحميل الأولي للنسخة الويب ما زال كبيرًا

النسخة المولدة للاختبار تقارب عدة ميغابايت بسبب تضمين المحتوى والأصول والطبقات القديمة. الإعداد `initial_payload_target_kb` هدف إنتاجي وليس نتيجة محققة بعد. الحل الفعلي:

- عدم تضمين بنك الأسئلة في HTML.
- تحميله على دفعات من API.
- CDN للأصول.
- Code splitting في Expo/Web.
- كاش محلي منظم.

### 4. SQLite لا يستوعب 100 ألف مستخدم كحل إنتاجي

SQLite مناسب للتطوير المحلي والفحص. إنتاج واسع يحتاج على الأقل:

- PostgreSQL مُدار.
- Redis للكاش والحدود والمهام.
- Object Storage/CDN.
- Load Balancer ونسخ API متعددة.
- Queue للإشعارات والمعالجة غير المتزامنة.
- مراقبة أخطاء وأداء وسجلات مركزية.

### 5. بنك الأسئلة غير جاهز للنشر

- العدد: 1000 سؤال تطوير.
- إصلاحات آلية: 18.
- المؤهل للنشر: 0.
- المصادر والحقوق غير موثقة بالكامل.
- الصعوبة غير معايرة على عينة طلاب حقيقية.

يلزم مسار مراجعة بشرية، ثم تجربة تجريبية وقياس أداء كل سؤال قبل استعماله لتوقع الجاهزية.

### 6. مؤشر الجاهزية ليس درجة رسمية

النظام الحالي مؤشر داخلي يعتمد على القياس وأداء الطالب داخل سهيل. لا توجد بيانات معايرة كافية لادعاء توقع رسمي. يجب ربط نتائج الاستخدام بدرجات طلاب حقيقية بموافقتهم، ثم معايرة النموذج والتحقق من الخطأ والثقة والإنصاف.

### 7. التحديات تحتاج نشر الخادم وربط العميل

API ينفذ طلبات الصداقة والقبول والتحدي والنتائج. لكن واجهة Streamlit تحتفظ بمعاينة محلية. التجربة الحقيقية تحتاج:

- عميل Expo يستخدم Bearer Token.
- Push Notification للدعوات والنتيجة.
- قواعد منع الغش وتوقيت خادم.
- اختيار أسئلة متكافئة ومعتمدة فقط.
- سياسات حظر وإبلاغ وخصوصية للقاصرين.

### 8. الإشعارات بعد إغلاق الجوال ليست فعالة بعد

المجلد `mobile/notifications` يحتوي أساس Expo، لكن التشغيل الفعلي يحتاج APNs/EAS، تسجيل التوكن، خادم إرسال، ساعات هادئة، وإلغاء الاشتراك.

### 9. الأمان يحتاج طبقة إنتاج إضافية

التحسينات الحالية جيدة كأساس، لكن الإنتاج يحتاج:

- HTTPS فقط.
- Secret Manager.
- تدوير وإلغاء التوكنات.
- Refresh Tokens أو جلسات قصيرة محكمة.
- Rate limiting موزع عبر Redis.
- تدقيق مدخلات شامل.
- CSP مناسبة للعميل.
- نسخ احتياطية واختبار استرجاع.
- سجل عمليات أدمن على الخادم لا في المتصفح فقط.

## قرار الجاهزية

الحزمة جاهزة كقاعدة **Sprint 54 للتطوير والاختبار الداخلي**، وليست جاهزة بعد للنشر العام أو استيعاب 100 ألف مستخدم. المرحلة التالية الصحيحة هي بناء عميل Expo وربطه ببيئة API/PostgreSQL اختبارية، بالتوازي مع اعتماد المحتوى، وليس إضافة طبقات أخرى داخل `app.py`.

# تقرير تنفيذ Sprints 40–47

## Sprint 40 — التنقل والتصميم

- بناء قائمة سفلية ثابتة مستقلة عن الصفحات.
- إبراز الصفحة الرئيسية بعنصر مركزي مرتفع.
- استبدال رموز النص بأيقونات SVG موحدة.
- إخفاء القائمة كليًا داخل الاختبار.
- داخل تفاصيل الملخص: استبدال القائمة بسهم يرفع لوحة التنقل.
- إعادة تصميم الرئيسية لتعرض الرحلة بدل الأرقام المنفصلة.

## Sprint 41 — إعدادات الأدمن العميقة

- لوحة إعدادات متعددة الأقسام.
- إعدادات عامة، تعلم، تشخيص، ستريك، تحديات، إشعارات، ميزات، محتوى وأداء.
- تصدير واستيراد JSON واستعادة الافتراضي.
- إضافة ملفات إعداد مركزية ومسار API محمي بتوكن.
- توثيق أدوار مدخل البيانات والمراجع والأدمن ومسار النشر.

## Sprint 42 — تحديد المستوى حسب الهدف والمسار

- ملف طالب يحفظ الهدف والمسار.
- قياس قدرات علمي 6/6، وأدبي 4/8.
- قياس تحصيلي علمي موزع على أربع مواد.
- نموذج توقع داخلي مختلف للعلمي والأدبي.
- تعطيل التحصيلي الأدبي بوضوح حتى وجود بنك معتمد بدل توليد نتيجة مضللة.

## Sprint 43 — الستريك والتحفيز

- نار للستريك.
- ساعة رملية عند بقاء 6 ساعات أو أقل وعدم وجود نشاط تعليمي في اليوم.
- عدم احتساب مجرد فتح التطبيق كنشاط.
- احتساب الإجابة، إنهاء الاختبار، تحديد المستوى وفتح ملخص.

## Sprint 44 — التحدي مع صديق

- صفحة تحديات مستقلة.
- كود صداقة لكل حساب.
- إضافة صديق، إنشاء تحدٍ، حالات التحدي ولوحة ترتيب.
- قوالب: يومي، سرعة، أسبوعي.
- جداول SQLite مبدئية للصداقات والتحديات.

## Sprint 45 — الشخصيات

- 8 شخصيات SVG خفيفة.
- 4 شخصيات ذكور و4 إناث.
- اختيار الشخصية من إعداد الرحلة.
- ظهور الشخصية في الرئيسية والحساب والتحديات.

## Sprint 46 — تلميع رحلة الطالب

- توحيد البطاقات والعناوين والأزرار.
- صفحة إعداد رحلة واضحة لأول طالب.
- ربط الهدف والمسار بالشاشة الرئيسية والتشخيص والتوقع.
- رسائل واضحة عند الميزات غير المكتملة، خصوصًا التحصيلي الأدبي.

## Sprint 47 — الأداء والتنظيم

- فصل CSS وJavaScript الخاصين بالمرحلة عن `app.py`.
- كاش لملفات الإعداد في Python.
- API بحدود صفحات وكاش HTTP قصير.
- أصول الشخصيات SVG خفيفة.
- إضافة مخططات منظمة للتكوين والدرجات والتحديات.
- توسيع فحص المشروع ليشمل JavaScript، الشخصيات، الإعدادات والجداول.

## النتيجة

الحزمة أصبحت نموذجًا تجريبيًا مترابطًا لرحلة الطالب، وليست مجرد مجموعة صفحات. ما يزال الانتقال إلى الإنتاج يحتاج Backend حقيقي، مصادقة، قاعدة بيانات مُدارة، إشعارات Expo/APNs ومراجعة المحتوى.

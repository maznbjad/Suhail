# Sprint 39 — البنية الحالية وحدودها قبل الإنتاج

## ما أصبحت الحزمة قادرة عليه

- تشغيل رحلة طالب كاملة بصورة محلية داخل متصفح Streamlit.
- حفظ التشخيص والأداء والأخطاء والخطة والإتقان في `localStorage` لكل حساب محلي.
- تحميل البيانات والأصول المستخدمة بكاش بدل إعادة قراءتها بلا داعٍ في كل إعادة تشغيل.
- تقديم بنك الأسئلة من SQLite عبر API قراءة مبدئي.
- تجهيز منطق إشعارات يمكن نقله إلى مشروع Expo.

## ما لا تدعيه هذه النسخة

### 1. ليست نسخة App Store أصلية بعد

واجهة Streamlit ما زالت نموذج الويب الحالي. النشر الأصلي يحتاج مشروع React Native + Expo، شاشات أصلية، EAS، إعدادات Apple، الخصوصية، والمراجعة على أجهزة فعلية.

### 2. ليست بنية 100 ألف مستخدم متزامن

وجود 100 ألف مستخدم خلال يوم يختلف عن وجودهم في اللحظة نفسها. الحزمة الحالية لا تحتوي Load Balancer أو Redis أو PostgreSQL مُدارًا أو CDN أو Autoscaling أو Queue أو Observability إنتاجية.

المسار الإنتاجي المقترح:

1. Expo client.
2. API خلف بوابة وLoad Balancer.
3. PostgreSQL مُدار مع Read Replicas عند الحاجة.
4. Redis للكاش والجلسات والحدود.
5. Object Storage + CDN للصور والصوت.
6. Queue للإشعارات والأعمال الثقيلة.
7. تسجيل مركزي للأخطاء والمقاييس والتنبيهات.
8. اختبارات تحميل تدريجية: 500، 2,000، 10,000 مستخدم متزامن قبل أي ادعاء أعلى.

### 3. البيانات الشخصية ما زالت محلية

الخطة والأخطاء والنتائج محفوظة في متصفح الجهاز. هذا مناسب للنموذج الأولي لكنه لا يزامن التقدم بين الأجهزة، وقد يفقد عند مسح بيانات المتصفح. يجب نقلها إلى حساب مستخدم وخادم آمن قبل الإطلاق.

### 4. إشعارات الجوال المغلق ليست مفعلة من Streamlit

تم تجهيز منطق Expo، لكن الإشعارات بعد إغلاق الجوال تحتاج APNs/Expo Push وخادمًا يحتفظ بالتوكن وجدول الرسائل. زر التذكير الحالي في واجهة الويب لا يضمن إشعارًا بعد إغلاق التطبيق.

### 5. بنك الأسئلة يحتاج تحريرًا بشريًا

الـ1000 سؤال هدفها تشغيل النظام وقياس الأداء والتغطية. لا تعد بنكًا رسميًا أو معتمدًا. قبل الاستخدام النهائي يجب:

- مراجعة كل سؤال والإجابة والشرح.
- إزالة التكرارات المعنوية.
- ضبط مستوى الصعوبة.
- ربطه بالمصدر والمنهج.
- التأكد من الحقوق وعدم نسخ محتوى محمي.
- تجربة الأسئلة على عينة طلاب وحساب صعوبة وتمييز كل سؤال.

## الدين التقني المتبقي

- `app.py` ما زال كبيرًا، رغم فصل الوحدات الجديدة.
- أجزاء قديمة من الواجهة ما زالت موجودة للحفاظ على التوافق، بينما تستبدلها طبقة Sprint 39 وقت التشغيل.
- يلزم لاحقًا نقل كل صفحة إلى مكوّن مستقل، ثم حذف النسخ القديمة بعد اختبارات انحدار.
- المصادقة الحالية ليست مصادقة إنتاجية.
- API القراءة الحالي لا يحتوي صلاحيات أو Rate Limiting أو Versioned migrations كاملة.

## بوابة الانتقال للسبرنت التالي

لا يبدأ تجهيز متجر Apple النهائي قبل إغلاق هذه البنود:

- اعتماد تصميم الرحلة بعد تجربة فعلية على 5–10 طلاب.
- تدقيق أول دفعة محتوى نهائي لا تقل عن 150 سؤالًا عالي الجودة.
- إنشاء مشروع Expo وتشغيله على iPhone عبر Expo Go/Development Build.
- نقل تسجيل الدخول والتقدم إلى API وقاعدة PostgreSQL.
- إضافة Analytics للأحداث التعليمية، وليس التصفح فقط.
- كتابة سياسة الخصوصية وشروط الاستخدام وآلية حذف الحساب.

### 6. تسجيل الدخول الحالي تجريبي وغير صالح للإنتاج

تحتوي النسخة المحلية على حسابات تجريبية وكلمات مرور محلية ويُنشئ التسجيل حسابات داخل `localStorage`. لا يجوز رفع هذا النظام كما هو للمتجر. قبل الإطلاق يجب إلغاء الحسابات الافتراضية، تخزين كلمات المرور بهش قوي على الخادم، إضافة استعادة حساب، تحقق بريد، Rate Limiting، وإدارة جلسات آمنة.
